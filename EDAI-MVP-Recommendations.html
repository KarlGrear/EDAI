Confirmed Gaps Fix Before Release

1
Off-Screen Window Restore Protection
High Priority
Location
EDAI.UI/App.xaml.cs lines 116–117
Effort
Low — ~4 lines of code
Risk if skipped
Users get stuck with an off-screen window and no obvious recovery path

App.xaml.cs unconditionally applies the saved WindowLeft and WindowTop values without checking whether that screen coordinate is still on a connected monitor. A user who plays on a dual-monitor setup and later disconnects the second monitor will relaunch EDAI to a blank desktop — the window is rendering off-screen.

Fix: Before applying the saved position, clamp it against SystemParameters.VirtualScreenLeft, VirtualScreenTop, VirtualScreenWidth, and VirtualScreenHeight. If the saved position falls outside the virtual desktop bounds, fall back to CenterOnPrimaryMonitor() (which is already implemented in App.xaml.cs).

2
Journal Directory Is Hardcoded — No Override in Settings
High Priority
Locations
EDAI.Core/Journal/JournalWatcher.cs lines 19–21
EDAI.Core/Journal/JournalAuxFileReader.cs lines 8–10
Effort
Medium — settings model, repository, UI field, service injection
Risk if skipped
Any non-standard install silently fails — app shows "journal directory not found" with no fix available

Both JournalWatcher and JournalAuxFileReader independently hardcode the path:

%USERPROFILE%\Saved Games\Frontier Developments\Elite Dangerous

Users who have Elite Dangerous installed on a non-default drive, who use NTFS junction links to redirect the save folder, or who need to point EDAI at a backup journal directory have no recourse.

Fix: Add a JournalPath field to SettingsModel and SettingEntity (with the current hardcoded value as the default). Expose it in the Settings window with a text box and a folder-browse button (FolderBrowserDialog). Inject the resolved path into both JournalWatcher and JournalAuxFileReader via their constructors rather than reading from a static field. Add a new EF Core migration for the column.

3
CLAUDE.md Contains Stale Namespace References
Medium Priority
Location
CLAUDE.md — Solution Structure section
Effort
Trivial — a global find and replace
Risk if skipped
Claude generates code with wrong namespaces (EDJP.* instead of EDAI.*)

The solution structure section in CLAUDE.md still lists the old project names EDJP.Core, EDJP.UI, and EDJP.Data. The actual projects use the EDAI.* namespace. Since CLAUDE.md is loaded as project instructions on every Claude Code session, this causes Claude to generate incorrect using directives, wrong project references, and incorrect file paths.

Fix: Global replace EDJP. → EDAI. in CLAUDE.md. Also update the directory names shown in the tree (currently EDJP.Core/, EDJP.UI/, EDJP.Data/).

4
No Unit Tests for Critical Pipeline Logic
High Priority
Scope
New EDAI.Tests xUnit project
Effort
Medium — ~30 test cases, no mocking required for the core targets
Risk if skipped
Edge cases in condition/template logic surface as user-reported bugs

The two most complex pieces of logic in the codebase — ConditionEvaluator and TemplateEngine — have zero test coverage. Both are pure, stateless functions that take strings and return strings or booleans. They require no mocking, no DI, and no UI setup to test.

Known areas of risk:

ConditionEvaluator — operator precedence (&& binding tighter than ||), mixed numeric/string comparisons, whitespace in token values, empty conditions, tokens that resolve to null.

TemplateEngine — nested JSONPath expressions (e.g. |trigger.Factions[0].Name|), count() wrapper with filter predicates, tokens in auxiliary files that are absent, hex-escaped characters in JSON values.

Also worth covering: JournalParser.TryParse() (malformed JSON, missing event field) and ResponseParser (AI returns partial schema, extra keys, invalid JSON).

Suggested setup: Add an EDAI.Tests xUnit project referencing EDAI.Core. No external dependencies needed beyond xunit and xunit.runner.visualstudio.

Medium Priority Nice for MVP Quality

5
Response Log Is Write-Only — No History Viewer
Medium Priority
Location
EDAI.Core/Interfaces/IResponseLogRepository.cs
EDAI.Data/Repositories/ResponseLogRepository.cs
Effort
Medium — repository query method + new UI tab or window
Risk if skipped
Users cannot audit what the AI said during a session they weren't actively watching

IResponseLogRepository exposes only AddAsync. Every AI response is persisted to the ResponseLogs SQLite table with the full prompt sent, raw AI reply, formatted display output, and announced text — but there is no way to retrieve or browse this data from within the application.

For a first pass, a minimal implementation would be:

1. Add GetRecentAsync(int count) to IResponseLogRepository and implement it with an OrderByDescending(r => r.Timestamp).Take(count) EF Core query.

2. Add a History tab to the main window (alongside the existing Responses and Event Log panels) that shows the last 100 entries — timestamp, config title, and displayed output — in a scrollable list. Clicking an entry could expand it to show the full prompt and raw AI response.

6
Empty State on the Configuration Selection Screen
Lower Priority
Location
EDAI.UI/Views/EventConfigSelectionWindow.xaml
Effort
Low — a few lines of XAML with a DataTrigger
Risk if skipped
Slightly poor first-run experience — blank grid with no guidance

A first-time user opens the Configurations window and sees an empty DataGrid with column headers and no rows. There is no hint that they should click New to get started.

Fix: Overlay a centered TextBlock with the text "No configurations yet. Click New in the toolbar to create your first pipeline." Use a DataTrigger on RowsView.Count == 0 to show/hide it. This is a common WPF pattern and takes about 10 lines of XAML.
